This specification defines the expected behaviors for for elements with the contenteditable attribute set to typing. This specification builds on HTML5.

Problem and Use Cases

The problems solved by and use cases for this document by this document are tracked in the Editing Explainer document.

Terminology

CE Typing Container
An element which has the contentEditable attribute set to `typing` which does not have it's css-display value set to `none` nor does it have any ancestors that do.
CE False Element
Any element that has the contentEditable attribute set to `false`.
CE Typing Node
Any descendant text node element of a CE Typing Container that does not have a ancestor element that is a CE False Element that is closer to it than the closest ancestor CE Typing Container nor does it have any ancestors that have their css-display value set to `none`.
Legal Caret Positions

All positions in which the caret can be placed programmatically. The placement is restricted to be in one of these positions:

  1. Before or after any character in any text node.
  2. After any inline element that is not a text node.
  3. Before any inline element that is not a text node and that does not have a `previousSibling`.
  4. Inside any empty inline or block element that is not a stub-element or inside any empty text node.
  5. Before or after a block element that is also a stub element.
  6. In the CE Typing Containeritself, if it is empty.

There are two exceptions to these rules:

  1. Within a table, the caret can only be placed inside `TH` and `TD` elements.
  2. The caret cannot be placed inside invisible nodes.

Introduction

This section is not normative

The typing state is the foundation of rich text editing on the web, and designed to be used only with a script-based editor implementation. It is not meant to be a stand-alone solution.

There is no default browser behavior for user input in contenteditable="typing". Intead, it fires Device-Independent Events that can be handled by scripts. Details on those events can be found in the Selection API and Input Events specifications. Input Events also contains information on default handling of text input inside a single CE Typing Node.

When the selections contains a CE Typing Container, `execCommand` and `queryCommand` may not work as they do in other editable elements, or may even fail to execute. These methods are not interoperable nor guaranteed to work against `contentEditable=typing`, and their uses are strongly discouraged.

Cutting and pasting

Within a CE Typing Container, cutting and pasting is disabled. However, `cut` and `paste` events are still triggered.

Caret drawing and movement

Caret drawing

A caret is drawn in any CE Typing Container that is focused, that does not hold any other selections and contains at least one CE Typing Node. Under such conditions, the caret represents a collapsed selection.

Caret positions

It MUST be possible to put the caret in any of the Legal Caret Positions programatically and for the caret to be visible in these.

Caret modes

inline
The caret is displayed as a line.
overwrite
The caret is displayed as an overlay over the character that follows it within the CE Typing Node. If the caret is at the end of a CE Typing Node, so that no character within the same node follows it, or it is placed before or after an element or inside an empty element or text node, it is displayed as a line, as if it were in inline-mode.

The caret is initially of type `insert`, but can be modified to be of type `overtype`. The caret of type `insert` is displayed as a line at the beginning of the text character node where it is placed. The caret of type `overtype` overlays the text node character where it is placed fully.

We need a method to switch from one type of caret to the other

Initial caret placement

Carets are initially placed at the first possible position within the CE Typing Container.

Caret movement

The caret is not moved by default, but `beforeInput` events are triggered when the user indicates an intention to move the caret.

Moving the caret within the line is not that difficult to do in JavaScript. Moving it in the block direction (or 180 degrees in the opposite direction) is much more difficult. It would help a lot if the browser could expose something about where the caret would be if it were to go one line into either direction. But what could the browser expose that would give us that?

Text Node Editing

When the user indicates that new content should be added to the CE Typing Node in which the caret is currently place, no content is actually added, but a `beforeInput` event is triggered with `inputType` set to `insertCharacter` or `overwriteCharacter`, depending on the current mode of the caret.

Adding characters by means of composition/IME

If a caret is present, and composition of a character through IME is initiated, composition visually is shown inline at the location of the caret. The composition takes place within the Shadow DOM, using the node the lowest level node the caret is contained in as a shadow root, with no changes occuring at the DOM level. If the caret is inside a CE Typing Node it acts as the `shadow root`. Otherwise the lowest level inline or block node acts as the shadow root. When the composition of the character(s) has finished, a `beforeInput` event is triggered for each generated character in the order they are to be added as if the character(s) had been entered via the keyboard.

If the caret is in overwrite-mode at the time the character composition is initiated, and the caret is not placed behind the last character within a CE Typing Node, the character that follows the caret is hidden during composition.

If the user cancels character composition before it has finished, the inline composition is cleared and the caret is returned to the state it had before composition was initiated.

Replacing text

When the user indicates the wish to replace part of the contents of a CE Typing Node by means of a browser-builtin spell checker or alike, a `beforeInput` input with `inputType` set to `replaceText` is triggered.

Removal of content

Content is not removed automatically through user input. Instead, Del/Backspace key presses, etc. trigger user `beforeInput` events with `inputType` set to `deleteContent`.

Context menu

If the implementation provides a `context menu`, this `context menu` SHOULD contain items for editing operations such as `paste`, `cut`, `copy`, `delete` and CAN contain items for spellchecking for any CE Typing Container. Triggering any of `context menu` items MUST NOT by default cause any change to the DOM, `beforepaste`, `beforecut` and beforeInput events SHOULD be triggered. Menu items in a `context menu` that do not cause DOM changes by default SHOULD function as in any other editing context.
